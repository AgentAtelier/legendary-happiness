# Visual-Quality Deep-Research Prompts (4 tools)

Four complementary deep-research prompts for raising Forge's visual quality (props, textures,
normal/bump maps, lighting, shadows, materials). Paste one block per tool. Differentiated so the
four reports complement rather than duplicate.

**How to run each:**
- **ChatGPT** — pick the **Deep Research** tool, paste the block. (If it asks to clarify, the block
  already pre-answers; reply "scope is as written.")
- **Mistral Le Chat** — toggle **Research** under the message box, paste the block, review/Edit the
  proposed step-plan, **Start research**.
- **Qwen Chat** — enable **Deep Research → Advanced Mode**, paste the block.
- **Local model** — paste as a normal chat prompt (no web; it's knowledge-only).

**Shared prioritization hypothesis** (each prompt asks the tool to validate/challenge it): the
biggest visual gains per unit effort are (1) procedural **PBR materials**, (2) **normal/bump maps**
to fake surface relief, (3) **triplanar** texturing (no UV unwrap), (4) a proper **interior lighting
rig**. Everything must stay **procedural, deterministic, single-GPU-local, Blender(bpy)→GLB→Godot 4.x
Forward+**.

## 1. ChatGPT — Deep Research (breadth + sourced synthesis)

```
You are a senior technical artist and real-time graphics engineer specializing in PROCEDURAL
content generation and the Godot 4 engine. Do deep research and produce a comprehensive, well-cited
report.

PROJECT CONTEXT. "Forge" is a generation-first engine that builds small playable 3D rooms entirely
procedurally: Python scripts drive Blender (bpy) to generate every prop as a .glb (~37 categories —
tables, chairs, shelves, etc.), a deterministic layout places them, and the scene runs in Godot 4.x
(Forward+ renderer). HARD CONSTRAINTS: (a) everything must be generated PROGRAMMATICALLY — no
hand-authored/painted textures, no manually sculpted meshes; (b) fully DETERMINISTIC (seed-stable
output); (c) must run on ONE consumer GPU locally; (d) builds are disposable. CURRENT STATE: props
are flat single-colour GLBs (no normal/roughness/AO maps), floors/walls/ceilings are flat-coloured
boxes, interior lighting is weak — the result reads like grey boxes.

RESEARCH MISSION. Find the current (2025–2026) best practices and concrete techniques to raise the
visual quality of a procedural Blender→Godot 4 pipeline, WITHIN the constraints above. Cover:
1. Procedural PBR materials (albedo/roughness/metallic/normal/AO): bake in Blender vs generate in
   Godot shaders — pros, cons, determinism, performance.
2. Normal & bump mapping: how to procedurally generate normal maps (from noise/height fields) to
   fake surface relief without extra geometry; how to apply/strength-tune them in Godot.
3. Procedural texturing WITHOUT UV unwrapping: triplanar mapping, noise (Perlin/Voronoi/Worley/FBM),
   weathering/dirt/edge-wear, per-instance variation from a seed.
4. Interior lighting in Godot 4: omni/spot/area rigs for enclosed rooms, emissive surfaces, and
   global illumination (SDFGI vs VoxelGI vs LightmapGI) — quality/cost/determinism tradeoffs.
5. Shadows: directional vs point shadows, soft/contact shadows, shadow atlas/bias tuning, common
   artifacts and fixes.
6. Architectural surfaces: good procedural floor/wall/ceiling materials (tiling, trims, baseboards).
7. Second-generation prop GEOMETRY: bpy techniques for richer silhouettes generated by script —
   bevels, modifiers, greebling/detailing, modular kits, LODs.
8. Post-processing: SSAO/SSIL, bloom, tonemapping (ACES/AgX), color grading, fog/volumetrics.
9. Godot integration & performance: GLB PBR import settings, material setup, Forward+ specifics,
   keeping generation fast and deterministic on a single GPU.

We HYPOTHESIZE the biggest wins are: procedural PBR + normal maps + triplanar + a real interior
lighting rig. Validate, challenge, or re-prioritize this.

OUTPUT. A structured report: one section per topic with concrete, actionable guidance (specific
Blender bpy APIs/node setups, Godot material/shader/GDScript specifics, parameter ranges, gotchas),
each backed by citations to docs/repos/articles. End with a PRIORITIZED ROADMAP table (technique →
expected visual gain → implementation effort → risk) ordered by gain-per-effort for a procedural,
deterministic, local pipeline. Prefer primary sources (Godot docs, Blender Python API, reputable
graphics blogs/GDC). Note any technique that breaks determinism or needs a second GPU.
```
